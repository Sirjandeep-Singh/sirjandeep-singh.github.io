Software projects inevitably grow in complexity. What starts as a few hundred lines of code can quickly become a large codebase maintained by multiple developers over months or years. As projects scale, the challenge is no longer just writing code that works, but ensuring that it remains readable, maintainable, and reliable.

The Code Quality lecture from Missing Semester focuses on a collection of tools and practices that help developers maintain high standards throughout the software development lifecycle. Rather than relying entirely on manual review, modern development workflows use automation to enforce consistency, catch bugs early, and verify correctness before code reaches production.

## Code Formatters

One of the simplest ways to improve a codebase is to standardize its formatting. Every developer has personal preferences regarding indentation, line wrapping, spacing, and code layout. While these differences may seem minor, inconsistent formatting makes code harder to read and often leads to unnecessary discussions during code review.

Formatters solve this problem by automatically applying a consistent style across an entire project. Instead of debating whether to use tabs or spaces or where to place line breaks, developers can let a formatter make those decisions automatically.

The biggest advantage of a formatter is that it removes subjective style discussions from the development process. Code reviews can then focus on architecture, correctness, and maintainability rather than formatting details.

Most modern editors can automatically run a formatter whenever a file is saved, making code formatting effectively effortless.

## Linters

While formatters focus on appearance, linters focus on code quality and correctness.

A linter analyzes source code without executing it and looks for suspicious patterns, potential bugs, and violations of project conventions. Many common mistakes can be detected long before a program is run.

Examples of issues a linter may detect include:

- Unused variables
- Dead code
- Incorrect imports
- Potential runtime errors
- Dangerous coding patterns
- Style violations

The lecture also introduces Semgrep, a tool that extends the idea of traditional text searching. Unlike grep, which searches plain text, Semgrep understands the structure of source code. This allows it to identify problematic code patterns even when formatting changes or code is spread across multiple lines.

This distinction highlights an important lesson: understanding code structure is often more powerful than searching raw text.

## Testing

Even perfectly formatted and linted code can still contain logical errors. Testing provides a way to verify that software behaves as expected.

Tests serve as executable specifications for a program. Instead of manually checking functionality every time code changes, developers can run automated tests to verify that existing behavior remains correct.

Testing offers several important benefits:

- Detects regressions
- Verifies correctness
- Documents expected behavior
- Enables safer refactoring

The lecture also discusses code coverage tools. Coverage reports measure which parts of a program are executed during testing. By examining coverage reports, developers can identify untested sections of code and improve the completeness of their test suites.

Coverage is not a perfect measure of software quality, but it provides useful insight into which parts of an application are actually being exercised by tests.

## Continuous Integration

As projects grow, manually running formatters, linters, and tests becomes increasingly unreliable. Developers forget steps, use different environments, or accidentally push code that breaks existing functionality.

Continuous Integration (CI) addresses this problem by automating quality checks whenever code is pushed to a repository or submitted through a pull request.

A CI pipeline commonly performs tasks such as:

- Running formatters
- Executing linters
- Running test suites
- Performing type checks
- Verifying builds

By automatically performing these checks, CI ensures that every change is evaluated using the same standards.

The lecture uses GitHub Actions as an example of a platform that can automate these workflows directly within a repository. Once configured, quality checks become part of the development process rather than something developers must remember to perform manually.

The result is a faster feedback loop and a more reliable codebase.

## Regular Expressions

The lecture concludes with an introduction to regular expressions (regex), a powerful language for describing text patterns. Regexes are commonly used for searching, validation, extraction, and text transformation tasks across editors, command-line tools, and programming languages.

Rather than searching for exact text, regex allows us to describe classes of patterns.

Some of the fundamental constructs introduced in the lecture include:

```text
abc
```

Matches the exact string `abc`.

```text
[abc]
```

Matches either `a`, `b`, or `c`.

```text
[^abc]
```

Matches any character except `a`, `b`, or `c`.

```text
[a-z]
```

Matches any lowercase letter from `a` to `z`.

```text
.
```

Matches any single character.

```text
*
```

Matches zero or more occurrences of the preceding pattern.

```text
+
```

Matches one or more occurrences of the preceding pattern.

```text
?
```

Matches zero or one occurrence of the preceding pattern.

```text
^
```

Matches the beginning of a line.

```text
$
```

Matches the end of a line.

Regex patterns can also be combined to describe increasingly sophisticated text structures. For example, quantifiers allow patterns to repeat, character classes define sets of acceptable characters, and anchors specify where matches may occur within a string.

One of the most useful aspects of regex is its portability. The same concepts appear in tools such as grep, sed, editors like VS Code and Vim, scripting languages such as Python, and many other software engineering tools. Learning a small set of regex operators unlocks powerful capabilities for searching, filtering, and manipulating large amounts of text.

At the same time, the lecture emphasizes that regex should be used appropriately. While it excels at pattern matching, complex structured data formats are often better handled using dedicated parsers rather than increasingly complicated regular expressions.
